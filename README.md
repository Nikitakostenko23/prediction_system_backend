# prediction_system_backend

prediction_system_backend - система прогнозирования медицинских исходов.  
Проект построен на Django с Celery для асинхронных задач и отдельным микросервисом на FastAPI, 
который выполняет предсказания с помощью обученных ML‑моделей (бинарная и многометочная классификация).

## О проекте

Система предназначена для медицинских учреждений. 
Основные роли - хирург и анестезиолог. Хирург создаёт медицинскую карту пациента 
и запускает прогноз, который уходит в очередь, обрабатывается Celery и ML‑сервисом, 
после чего результат возвращается и отображается в интерфейсе. 
У анестезиолога нет прав на создание прогнозов, но доступен их просмотр.

Также выделяются роли ML‑разработчика, который имеет возможности хирурга, 
а также может добавлять в систему новые модели машинного обучения 
и сводную информацию о них. Администратор может всё, что предыдущая роль, 
а также управляет правами доступа пользователей.

Подробные user‑сценарии можно посмотреть в `docs/manual_scenarios.md`.

## Медицинская задача

Система предназначена для поддержки принятия врачебных решений 
при четырёх типах оперативных вмешательств:

- **КАС** - каротидная ангиопластика со стентированием
- **КЭЭ классическая**
- **КЭЭ эверсионная**
- **КЭЭ гломуссохраняющая**

На основе клинических и инструментальных данных сервис прогнозирует 
риск послеоперационных осложнений (инсульт, ТИА, тромбоз, пневмония и др.).

## Основные возможности

- Создание и управление медицинскими картами.
- Запуск прогнозов на фоне (асинхронная обработка).
- Отмена задач, находящихся в очереди или процессе выполнения.
- Детальный просмотр результатов прогноза, 
которые представляют собой вероятности для бинарных и многометочных моделей.
- Разграничение ролей и прав доступа:
  - хирург
  - анестезиолог 
  - ML‑инженер 
  - администратор.
- Очереди через Celery, в качестве брокера выступает RabbitMQ.
- Периодические задачи через Celery Beat (например, очистка старых прогнозов).
- REST API для взаимодействия с ML‑сервисом (FastAPI + Uvicorn).

## Архитектура системы

Система состоит из следующих компонентов:

1. **Django (бэкенд + админка)** - управление пользователями, медицинскими картами, правами доступа и задачами на прогноз.
2. **PostgreSQL** - основная база данных (пользователи, карты, результаты прогнозов, статусы задач).
3. **Redis** - используется как бэкенд для Celery (хранение состояния задач) и как кэш.
4. **RabbitMQ** - брокер сообщений между Django и Celery.
5. **Celery Worker** - выполняет асинхронные задачи: отправка запроса в ML‑сервис, сохранение результата, обработка отмен.
6. **Celery Beat** - планировщик периодических задач (например, автоматическое удаление старых записей 
или повторные попытки неудавшихся прогнозов).
7. **ML‑микросервис (FastAPI)** - загружает предобученные модели и возвращает предсказания.

Взаимодействие:

- Хирург через веб‑интерфейс Django создаёт карту пациента и запускает прогноз.
- Django формирует Celery‑задачу и отправляет в RabbitMQ.
- Celery‑воркер забирает задачу, формирует запрос к FastAPI и получает результат.
- Результат сохраняется в PostgreSQL и становится доступным для просмотра.
- Celery Beat периодически запускает задачи по расписанию (например, очистку неактуальных прогнозов).

## Реализация ML‑микросервиса

ML‑микросервис (папка `ml_service`) написан на FastAPI и использует следующие внутренние модули:

- **`ml_interface/estimator.py`** - базовые классы `BaseBinaryModel` и `BaseMultilabelModel`. 
Любая конкретная модель (для КАС, КЭЭ и т.д.) наследуется от одного из них и реализует методы `load()` и `predict()`. 
Это обеспечивает единообразный интерфейс для вызывающего кода.
- **`ml_interface/convertor.py`** - преобразует Pydantic‑модель медицинской карты (пришедшую от Django) в `pandas.DataFrame`, 
пригодный для подачи в модель.
- **`ml_interface/model_registry.py`** - реестр всех доступных моделей. 
Позволяет получить нужную модель по строковому идентификатору (например, `cas_binary` или `cee_classic_multilabel`).

Сам FastAPI предоставляет два эндпоинта:
- `POST /predict/binary/` - бинарный прогноз (наличие осложнений - 0/1)
- `POST /predict/multi_label/` - многометочный прогноз по группам осложнений

При старте микросервис загружает все модели из папки `ml_service/ml_interface/models/` в реестр, 
поэтому первый запрос может быть чуть медленнее, зато последующие работают быстро.

## Методы машинного обучения

В рамках проекта проведены эксперименты по двум задачам:

- **Бинарная классификация** - риск возникновения осложнения (0/1)
- **Многометочная классификация** - одновременный прогноз по всем 3-м группам осложнений

### Используемые библиотеки

- `pandas`, `numpy`, `matplotlib`, `seaborn` (визуализация, преодобработка)
- `scikit-learn` (предобработка, метрики, валидация, подбор гиперпараметров, модели)
- `xgboost`
- `keras` (MLP)
- `imblearn` (SMOTE)

### Валидация и метрики

- Стратифицированная кросс-валидация (5 фолдов, split 70/30)
- Для нейронных сетей: разбиение 70/15/15 с аугментацией и SMOTE
- Метрики: Accuracy, Recall, F1-score (macro/weighted), ROC-AUC, MCC

### Загруженные модели

Для каждого типа операции и задачи прогнозирования подготовлены предобученные модели (сохранены в `ml_interface/models/`):
- `cas` (КАС)
- `cee_classic` (КЭЭ классическая)
- `cee_eversional` (КЭЭ эверсионная)
- `cee_preserve_glomus` (КЭЭ гломуссохраняющая)

Каждая модель включает:
- Предобработчик (`*_preprocessor.pkl`)
- Редуктор размерности (PCA/LDA, `*_lda.pkl` или `*_pca.pkl`)
- Классификатор (файл `.keras` для нейронной сети или `.pkl` для sklearn‑модели)

## Связь с другими репозиториями

- **`model_training_for_prediction_service`** - исследовательский репозиторий. 
В нём находятся Jupyter‑ноутбуки с экспериментами, подбором гиперпараметров, визуализацией метрик 
и черновыми версиями моделей.
- **`prediction_system_backend`** - production‑репозиторий. 
Здесь перенесены только лучшие, проверенные модели (артефакты `.pkl`/`.keras`) и уже реализован сервис с чёткими API, 
готовый к деплою.

Такое разделение позволяет изолировать исследовательскую стадию от боевого кода и упрощает развёртывание.
